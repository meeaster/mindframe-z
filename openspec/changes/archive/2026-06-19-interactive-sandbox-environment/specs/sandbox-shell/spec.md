## ADDED Requirements

### Requirement: Interactive shell environment in the container

The sandbox image SHALL provide an interactive shell environment built on zsh,
oh-my-zsh, powerlevel10k, and mise so an operator can enter the container and work
in a shell that approximates their WSL shell. Entering the sandbox without naming
an agent SHALL drop the operator into this interactive shell.

#### Scenario: Operator enters an interactive shell

- **WHEN** the operator launches the sandbox without specifying an agent command
- **THEN** the container starts an interactive zsh session with the powerlevel10k
  prompt active

#### Scenario: Agents are available as PATH commands inside the shell

- **WHEN** the operator runs `claude`, `opencode`, or `gh` from inside the
  interactive shell
- **THEN** each command starts using the same brokered environment that a direct
  launch would receive

### Requirement: Dev dependencies installed via mise at image build

The sandbox SHALL manage container dev dependencies with mise from a committed
`mise.toml`, and the image build SHALL run `mise install` so declared tools are
already present before container startup.

#### Scenario: Declared tools are present after start

- **WHEN** the container starts from the built image
- **THEN** the tools declared in `mise.toml` are installed and available on the
  shell PATH

#### Scenario: Tool versions come from the committed manifest

- **WHEN** `mise.toml` declares a tool version such as Node 24
- **THEN** the installed tool resolves from the manifest at image build rather
  than from an ambient host version

#### Scenario: New releases are gated before install

- **WHEN** the image build resolves a fuzzy tool version from `mise.toml`
- **THEN** mise applies the configured minimum release age before selecting an
  installable release

### Requirement: Shell config is sanitized and non-secret

The sandbox SHALL lay down only sanitized shell configuration (`.zshrc`,
`.p10k.zsh`, `mise.toml`) from non-secret sources committed to or generated by
the sandbox. It MUST NOT mount the host home directory or host secret-bearing
dotfiles, and it MUST NOT introduce persisted shell history or rc files that
export real credentials.

#### Scenario: Host home and secret dotfiles are not mounted

- **WHEN** container mounts are enumerated after launch
- **THEN** the host `$HOME` and host dotfiles such as `~/.zshrc`,
  `~/.zsh_history`, and `~/.p10k.zsh` are absent

#### Scenario: Laid-down config contains no real credentials

- **WHEN** the in-container `.zshrc`, `.p10k.zsh`, and `mise.toml` are inspected
- **THEN** they contain no real provider credential or token, only prompt, tool,
  and shell configuration

### Requirement: Shell config is a replaceable stand-in

The sandbox shell configuration SHALL be a self-contained stand-in that does not
hardcode operator-specific host paths, so it can later be replaced by an external
profile source without changing the runner.

#### Scenario: No personal host paths are baked in

- **WHEN** the shell configuration and image build inputs are inspected
- **THEN** they reference no operator-specific absolute host path
