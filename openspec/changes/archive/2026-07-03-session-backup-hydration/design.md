## Context

mfz already manages cross-session thread logs (`mfz thread`) whose synthesized store is backed up to git-remote `destinations`. The *raw* session transcripts those threads are built from live only on the machine, in the harness stores (`~/.claude/projects/*/<id>.jsonl`; OpenCode's per-session JSON + `opencode.db`). Claude Code deletes transcripts at `cleanupPeriodDays` (~30 days), so long-lived threads lose their source and there is no durable, out-of-git home for raw history (too large, contains secrets).

A prototype (branch `feat/sessions-backup-prototype`, commit `76f5f41`) de-risked the design: a real `mfz sessions backup` slice ran against a live bucket (118 sessions uploaded, then 117 skipped / 1 churned on re-run), and a throwaway lifecycle model (`src/sessions/prototype/`) validated the `local → backed-up → vanished → hydrated` flow and its edge cases. This change turns that into the shipped feature. Prototype notes: `src/sessions/prototype/NOTES.md` (on the prototype branch; removed by this change's cleanup step).

## Goals / Non-Goals

**Goals:**
- A `mfz sessions backup` command that idempotently mirrors all local Claude Code + OpenCode sessions to an S3 archive, full-fidelity, keyed by id.
- An `archives` machine-config concept with one default-writable archive and read-only sources, resolved like thread `destinations`.
- Threads can hydrate a vanished session back from a readable archive on refresh/ingest, transparently and by prefix.

**Non-Goals (deferred, design-compatible):**
- Scheduling (cron/launchd/systemd) — run the command manually or wire external cron.
- Archive **discovery** and its metadata `index.jsonl` — no need to search past 30 days yet; this is the only deferred item that would reach back into backup, so deferring it keeps backup lean.
- Cross-machine watermark **pull**, work→personal **promotion**, and profile-pinned archive credentials (`profile` set → hard error until `@aws-sdk/credential-provider-ini` is wired).

## Decisions

**Archives as a first-class machine-config concept (sibling to `destinations`), not embedded in destinations.** One home for S3 config keeps it DRY; a destination could later *reference* an archive by name. Routing is by **origin machine**, and the read/write boundary is enforced by IAM credentials, not code — a machine that lacks work write-creds *cannot* leak a work session into a personal-readable bucket. Alternative (route per-thread/destination) was rejected: a work-machine session bound to a personal thread would leak work-sensitive data; sensitivity follows the machine a session ran on, not its intended category.

**Full-fidelity, self-describing per-session artifacts; bucket-is-the-catalog.** Claude `.jsonl` is copied verbatim (already self-describing, per-session). OpenCode uses the vendor-maintained `opencode export <id>` (`{ info, messages }` JSON) rather than reading its store schema directly — the export absorbs OpenCode's own migrations, so we never track their schema and archived artifacts stay readable regardless of future changes. Flat `<prefix>/<harness>/<id>` keys mean `ListObjectsV2` is the catalog; no local index to drift. Alternatives rejected: `aws s3 sync`/rclone (user prefers the official SDK where one exists); a local manifest index (second source of truth); sanitized exports (lossy — the boundary is a private, encrypted, IAM-scoped bucket instead).

**Incremental freshness guard (source signal vs stored `LastModified`, margin-adjusted).** One paginated `ListObjectsV2` builds `key → LastModified`; upload only new/newer. `LastModified` is upload time — always later than the write that triggered it — so a write landing mid-sweep would compare older-than-stored and be frozen out of every future run; the guard therefore uploads when the source signal is newer than `LastModified` minus a fixed safety margin, costing at most one redundant upload per session (the re-upload pushes `LastModified` far past the frozen signal, so it self-terminates). Claude's signal is transcript mtime; OpenCode's is derived entirely from `opencode.db` — the greater of the session row's `time_updated` and the latest message `time_created` for that session. (Current OpenCode keeps session info in the `session` db table, not a `storage/session/info/<id>.json` file — that path is a stale artifact of an older store layout; sessions are enumerated with `SELECT id FROM session`. Message growth may not bump `session.time_updated`, so folding in the latest message time keeps active sessions from being silently skipped.) Per-run cost tracks churn — load-bearing because `opencode export` is a process spawn per session. Mutable-session correctness falls out: an appended transcript bumps the signal → re-uploaded.

**Subagent transcripts stored under the session prefix (prototype decision).** `<id>/subagents/agent-*.jsonl` are backed up at `claude-code/<id>/subagents/*`, never as standalone ledger entries — asymmetric regret (cheap to keep, irreversible after deletion) plus the explicit full-fidelity principle. **Ripple:** hydration pulls a session by `ListObjectsV2` on the `<harness>/<id>` prefix, then `GetObject` each — not a single key. The archive-cache layout preserves the subtree.

**Hydration writes to a read-only cache, not the live store.** `~/.mindframe-z/archive-cache/` (gitignored, RO-mounted into the tools container). The live store is never mutated — the harness can't re-delete hydrated files, read-only-archive provenance is preserved, and the format differs anyway (OpenCode export JSON ≠ native store). The cache is write-once per session: an existing cached copy is used without re-consulting the archive (refreshing it from a newer archive copy belongs with the deferred cross-machine pull). Hydration fires only for sessions absent from the host store — a shrank-but-present session is left untouched with no archive consult. Gather reads via the existing explicit-path seam (`locateClaudeTranscript` already hands gather a path); generalize that seam to OpenCode and teach the `opencode-sessions` skill to read archived export JSON. Watermark adds one source-fallback (`readWatermark`: local store else cache) plus `tailSignatureFromExport` for the OpenCode archived form — mirroring `tailSignatureFromJsonl`.

**Atomic per-session hydration, not best-effort.** A multi-object session download (transcript + subagent transcripts) stages into `<cacheRoot>/.staging/<harness>-<uuid>/` and is committed via `rename` only once every object succeeds, with a `finally` that always clears the staging dir. Without this, a mid-download failure (e.g. one subagent transcript unreachable) could leave a primary-only cache that `isHydrated` reports complete forever — a partial, permanently-stuck false positive. Rejected: fixing up a partial cache in place, since detecting "this cache is incomplete" after the fact requires re-deriving what *should* be there — the same information the staging step already has for free.

**Negative caching, gated on genuine confirmation.** A session confirmed absent from every reachable, non-pinned archive gets a sticky `<id>.absent` marker so repeat sweeps over long-vanished sessions don't re-pay a full archive query. The gate is deliberately narrow: the marker is written only when at least one archive was actually queried (`checkedAny`) and every queried archive returned zero objects with no error (`allConfirmedAbsent`). An unreachable archive, a download error, or an all-profile-pinned archive set must never write the marker — any of those would turn a transient or credentialed-out condition into a permanent false negative that no future retry could undo without manual cache cleanup.

**Profile-pinned archives are skipped, not errored, on the read path.** The write path (`mfz sessions backup`) hard-errors on a profile-pinned default archive (see Non-Goals) because there is exactly one write target and silently using the wrong credentials would risk a misdirected upload. Hydration instead reads across every *configured* read-only archive, so a profile-pinned entry is simply skipped — the read simply doesn't happen for that source, sibling archives are unaffected — rather than raising an error that would abort hydration for an otherwise-recoverable session. This closes an asymmetry found in review: the original hydration code would have constructed an `S3Client` for a profile-pinned archive using ambient default-chain credentials, silently reading with the wrong identity instead of skipping or erroring.

**OpenCode data-home resolution is centralized, not duplicated.** Both session enumeration/export (`opencode-source.ts`) and watermark db reads (`watermark.ts`) resolve OpenCode's data directory the same way — `opencodeDataHome`/`opencodeDbPath` in `src/core/paths.ts`, honoring `XDG_DATA_HOME` with the `~/.local/share` fallback, matching OpenCode's own `xdg-basedir`-based resolution. Before this, each call site hardcoded `<home>/.local/share/opencode`, which is silently correct only when `MFZ_HOME`/`--home` is never used; under an override the two call sites would disagree about which db the enumeration and the `opencode export` shell-out were reading. Centralizing the resolution — and passing the same `XDG_DATA_HOME` through to the `execa` env for the export call — makes both paths agree by construction instead of by convention.

**Accept the `stale-recover` edge with no code.** Hydration can return an id whose archived tail is *older* than the ledger cursor (final edits never backed up before deletion). Since a read-only consumer machine cannot back up (IAM), "backup-then-hydrate in refresh" is impossible; the mitigation is operational (regular backup = the deferred scheduling item). Treat as `vanished`/untouched, with at most a one-line warning when the hydration consumer lands.

## Risks / Trade-offs

- **Secrets in the bucket** → Private bucket, public access blocked, IAM-scoped, and every `PutObject` requests `ServerSideEncryption: AES256` explicitly (objects stay encrypted even if bucket-default encryption is missing); full fidelity is deliberate. Never committed; `archives` live only in machine-local `~/.mindframe-z/config.yml`. Bucket-hardening is an out-of-band operator prerequisite, but because a misconfigured bucket would leak full-fidelity secrets, the command adds a **preflight** (`GetPublicAccessBlock`, all four flags true) that aborts before uploading anything if public access is not fully blocked — one API call of defense-in-depth against pointing at the wrong bucket. Bucket policy and IAM remain operator-owned.
- **`stale-recover` data gap** → Accepted; operational mitigation via backup cadence. The deletion clock is effectively an inactivity clock, so any backup in the idle window captures final state. (Note: Claude's exact deletion trigger is not documented explicitly — inferred as mtime-based; verify against CC source if it ever matters.)
- **`opencode export` cost per session** → Bounded by the freshness guard to churn only, so a daily sweep re-exports only changed sessions.
- **New dependency `@aws-sdk/client-s3`** → Standard vendor SDK; creds via default provider chain so mfz stores no secret. Pin exact version, >7 days old.
- **Profile-pinned creds unsupported** → Hard, clear error rather than silent wrong-account writes; wire `@aws-sdk/credential-provider-ini` when a real profile-pinned archive appears.

## Migration Plan

Additive, no breaking changes; existing thread `destinations` are untouched. Steps: (1) land `archives` schema + `mfz sessions backup` (Claude already prototyped, add OpenCode); (2) add hydration + archive-cache and wire the `vanished` fallback in refresh/ingest; (3) add `opencode-sessions` skill guidance + `tailSignatureFromExport`; (4) gitignore + RO-mount the cache; (5) docs. Rollback is removing the `archives` block and the command — no persisted local state beyond the disposable cache and the S3 objects. Tear down the scratch prototype bucket (`mfz-sessions-proto-*`, commands in `NOTES.md`) once real usage supersedes it.
