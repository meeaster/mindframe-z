## ADDED Requirements

### Requirement: Machine credential mode selects the Claude broker leg

The sandbox SHALL select the Claude Code credential leg from the machine's sandbox
credential mode declared in machine config (`~/.mindframe-z/config.yml`). The mode
SHALL be one of Bedrock or subscription. When machine config does not declare a mode
explicitly, mindframe-z MAY auto-detect it from the machine-local Claude settings, and
an explicit machine-config value SHALL take precedence over detection. The profile
manifests SHALL remain identical across machines regardless of the selected mode.

#### Scenario: Bedrock mode selects the signer leg

- **WHEN** machine config selects the Bedrock credential mode
- **THEN** the sandbox uses the Bedrock signing proxy leg for Claude Code

#### Scenario: Subscription mode selects the brokered subscription leg

- **WHEN** machine config selects the subscription credential mode
- **THEN** the sandbox uses the Agent Vault-brokered Claude subscription leg for Claude
  Code and does not start the Bedrock signer

#### Scenario: Explicit mode overrides detection

- **WHEN** machine config explicitly declares a credential mode and the machine-local
  Claude settings imply a different one
- **THEN** the explicitly declared mode is used

#### Scenario: Profiles are unchanged across modes

- **WHEN** the same profile is used on a Bedrock machine and a subscription machine
- **THEN** the profile manifests are byte-identical and only machine config differs

### Requirement: Init generates and persists broker initialization state

`mfz sandbox init` SHALL generate a strong Agent Vault master password and a scoped
agent token (instance role `no-access`, vault grant `local-ai-dev-sandbox:proxy`), start
Agent Vault so its persistent data volume is created, create the `local-ai-dev-sandbox`
vault, and fetch the MITM CA. Generated secret values SHALL be persisted only to the
machine-local secrets location (`~/.mindframe-z/secrets/`) with restricted permissions
and SHALL NOT be committed or printed. Init SHALL NOT overwrite an existing master
password or broker state, and SHALL direct the operator to back up the machine-local
secrets file because it is the sole copy of the recovery root. Provider credentials
(OpenAI, GitHub, Bedrock/AWS, Claude subscription) SHALL NOT be generated by init.

#### Scenario: Master password is generated and stored privately

- **WHEN** init runs on an uninitialized machine
- **THEN** a strong Agent Vault master password is generated and stored under
  `~/.mindframe-z/secrets/` with restricted permissions and is not printed

#### Scenario: Existing master password is never overwritten

- **WHEN** init runs and a master password already exists in the machine-local secrets
- **THEN** init leaves it unchanged and does not regenerate or overwrite it

#### Scenario: Scoped agent token is generated

- **WHEN** init creates the sandbox vault
- **THEN** it generates an agent token with instance role `no-access` and vault grant
  `local-ai-dev-sandbox:proxy` and persists it to the machine-local secrets location

#### Scenario: Data volume and vault are created

- **WHEN** init starts Agent Vault for the first time
- **THEN** the persistent broker data volume and the `local-ai-dev-sandbox` vault exist

#### Scenario: Operator is told to back up the recovery root

- **WHEN** init completes a fresh initialization
- **THEN** it instructs the operator to back up the machine-local secrets file without
  printing the master password value

#### Scenario: Provider credentials are not fabricated

- **WHEN** init completes
- **THEN** no real provider credential has been generated, and provider credential
  seeding remains a separate step

### Requirement: Agent Vault operational secrets have a defined machine-local source

The sandbox SHALL source Agent Vault operational values (such as the Agent Vault agent
token and the broker master password) from a defined machine-local location outside the
repository, following the existing machine-local secrets convention
(`~/.mindframe-z/secrets/`). These values SHALL NOT be committed and SHALL NOT be
required as committed `sandbox/.env` content for clone-and-run.

#### Scenario: Operational secrets load from machine-local secrets

- **WHEN** `mfz sandbox` needs the Agent Vault agent token or broker master password
- **THEN** it reads them from the machine-local secrets location and not from a
  committed file

#### Scenario: Clone-and-run does not require committed secrets

- **WHEN** the repository is freshly cloned and the machine-local secrets are present
- **THEN** `mfz sandbox` runs without any secret values committed to the repository

### Requirement: Broker Claude subscription auth for Claude Code

The sandbox SHALL support Claude Code with Claude subscription auth brokered by Agent
Vault when the machine credential mode is subscription. The agent container MAY contain
placeholder Claude subscription auth values, but the real subscription OAuth access
token and refresh token MUST remain in Agent Vault storage and be injected and refreshed
only by the broker.

#### Scenario: Subscription request is brokered

- **WHEN** Claude Code sends a request with placeholder subscription auth and the
  machine credential mode is subscription
- **THEN** Agent Vault injects the real Claude subscription credential before forwarding
  the request

#### Scenario: Subscription OAuth credential is refreshed by broker

- **WHEN** the configured Claude subscription credential is OAuth-managed and its access
  token nears expiry
- **THEN** Agent Vault refreshes the token and injects the fresh credential without
  exposing it to the agent container
