# Ingesting Sessions Into a Thread-log

Discovery and extraction for the [thread-log](SKILL.md) pipeline. The **how** of reading each store lives in [`agent-sessions`](../agent-sessions/SKILL.md) — load it with the Skill tool (not by Read-ing its `SKILL.md`) and use the Claude Code or OpenCode branch. This file covers only what is specific to building a thread.

## Discover (plan step 1)

Goal: a deduped candidate set spanning both tools. **Delegate this to a Haiku Explorer subagent** — it is mechanical legwork, not judgment, so it does not belong in the capable parent. The Explorer **retrieves and over-collects**; it does **not** score relevance or filter (that is the approval gate's job, with the user). Match broadly — over-collecting is fine; missing a session is not. The Explorer's prompt must tell it to:

1. Derive topic terms from the charter (Update) or the user's description of the work (Create). A thread usually has several subtopics named inconsistently across sessions — cast a wide net with synonyms and component names.
2. Run `agent-sessions` "find a session" recipes for **both** branches (loading the skill via the Skill tool), but topic-scoped rather than project-scoped: filter Claude Code's transcript index and search OpenCode's `session` titles on those terms.
3. Union the results, dedupe Claude Code candidates to one per `sessionId` (keep the earliest prompt as its label), and return the set as text — source, id, title/first-prompt, project — with no relevance verdict.

## Confirm (plan step 2)

Present **only** candidates minus `sessions[]` minus `excluded[]` — the set never ruled on. Score each against the charter and show: source, id, title/first-prompt, project, and a one-line relevance hint (e.g. *"matches: CJIS/SDS redaction"* vs *"weak: cost work, likely out of scope"*). **Relevance scoring is yours, not the Explorer's** — it is the judgment the cheap retrieval deliberately left undone. The user rules on each; write the result to the ledger so it never resurfaces.

## Extract — gather, then synthesize (worker step 4)

Extraction is **two-stage, one session at a time** — never batch sessions (it flattens the detail, which is the whole failure this skill exists to prevent). The split exists because the two halves of extraction want different models: **reading** the whole transcript is token-heavy but needs little judgment, while turning what was read into faithful buckets — preserving the *why*, the rejected alternatives, the user's thinking-aloud — is pure judgment. So a cheap **gatherer** reads; a capable **synthesizer** judges. The gatherer is the only thing that touches the raw transcript; the synthesizer reads only the gatherer's dossier, never the transcript, which is what keeps the capable (expensive) model's cost proportional to *thinking*, not *reading*.

**Pair the stages per session.** Each session's dossier feeds only its own synthesis — dossiers are never pooled into one synthesizer, so running many captures at once stays bounded (one ~10–25k-token dossier in flight per session, not the sum of all of them).

### Stage 1 — gather (cheap model, e.g. Haiku)

Spawn **one Explore subagent per qualifying session** as the gatherer. It is read-only and **returns its dossier as text** — it has no Write tool and must never be told to write anything (told to write under `~/.claude` it refuses or improvises a fragile heredoc). Its job is mostly **retrieval, not judgment**: surface the **human record** — every user message, fork, decision, and the *why* — **verbatim and over-included**, because compressing *that* is the synthesizer's job and anything dropped here is lost before judgment ever runs. The one exception is **tool/MCP output**: bulky and mostly noise, it is the gatherer's to **read in full and distill to the finding that mattered** — the cheap judgment worth doing here, since raw tool dumps are the bulk of a transcript and forwarding them whole to the synthesizer is the token cost this stage exists to cut. Its prompt must:

1. Tell it to **load** `agent-sessions` **via the Skill tool, not by Read-ing the SKILL.md file**, then use the Claude Code or OpenCode branch; the skill owns the read-only, outline-first storage and drill-down recipes. "Read and follow the file" produces a file-Read and skips the loading mechanism, leaving no `Skill` tool_use for a `read_subagents` thread to detect.
2. Give the exact locator: project + `sessionId` (Claude Code) or `id` + directory (OpenCode). For an Update where the session has a `high_water`, gather only content past that offset.
3. Demand **verbatim** evidence, each item with a `[YYYY-MM-DD HH:MM]` timestamp (the record's own time — Claude Code record timestamp, OpenCode `time_created`) and a turn/part locator, grouped so the synthesizer can find each kind: **every user message quoted in full** (do not paraphrase, truncate, or clean up rambling — the user's thinking-aloud is the highest-value, most-easily-lost material); **every design fork / binary choice posed and its answer**, both verbatim (this is where rejected alternatives live); **every live tool/MCP test in sequence** (what was proposed → tested → returned → decided), with the **result distilled to the finding** — keep what was learned or settled, drop the raw payload (logs, dumps, long file bodies); **every decision and its stated reason**; and **anything said about why / motivation / vision**, even fragments.
4. **Return the whole dossier as one text message** — never write it to a file, and never hand back a path or a summary in place of the evidence (a summary editorializes, and the synthesizer then distrusts it and re-reads the source — the exact cost this split exists to avoid). If the dossier is too large to return inline, the **harness itself spills it to a `tool-results/*.txt` file** and the synthesizer reads that file — that is the dossier, and it is the synthesizer's one legitimate Read. The gatherer never makes that decision; it just returns everything and lets the harness handle the overflow. For the human record, completeness over brevity always; for tool output, the distilled finding over the raw dump.

**Subagent depth** is the gatherer's, and the manifest's **`read_subagents`** flag is the thread's **floor**: `true` (meta / process / skill-usage threads) gathers every session **deep** — the full nested subagent transcripts (`subagents/agent-*.jsonl`), reporting which **skills each subagent loaded** (`Skill` tool_use), its tool sequence, and where it diverged; `false` (default, domain threads) gathers **shallow** — each subagent's input and output from the parent transcript, which is enough for domain work. But a session goes deep regardless of the floor when it **exercises a skill** — building, testing, or using one. *A skill is a codified process, and its subagents enact it*: when the session is about the skill, the subagents **are** the subject, even inside a domain thread (the `datadog-cost` thread's cost-report session, run via a skill, is the case). This is **not** a per-run judgment — it is an observable transcript fact: a `Skill` tool_use, or a `SKILL.md` created or edited. Same session resolves to the same depth every run. Pass the resolved depth (floor OR exercises-a-skill) into the gatherer's prompt; the session skill's subagent-tracing recipes cover both.

### Stage 2 — synthesize (capable model — the parent)

You, the worker process (the synthesizer — whatever `--model` you were launched with), read **only the dossier** and write `sessions/<id>.md` to the contract in [ARTIFACTS.md](ARTIFACTS.md). Keeping the write in the worker, not the gatherer, makes the store strictly read-only to the gatherer and preserves token isolation. Turn the dossier into the bucket schema — **structured extraction, not free-form summary** — every bullet timestamped and carrying a session-id-qualified citation. For a contested decision, capture the **fork that was posed and why the chosen path won**, not just the settled outcome; for the Intent & Vision bucket, **prefer the gatherer's verbatim user quotes** and never invent phrasing for the user's voice — when you compress the user's words, you flatten exactly what this skill exists to keep. On an Update, append the new entries to the existing `sessions/<id>.md` rather than rewriting it.

**Never open the raw transcript yourself.** The dossier (the gatherer's text, or the `tool-results/*.txt` file the harness spilled it to) is your **only** source — reading the session `.jsonl` directly is exactly the token cost this split exists to avoid, and re-billed on every turn after, it is the single largest way a worker run blows its budget. If the dossier looks contradictory, thin, or missing a fork, that is a **gatherer** failure: **re-spawn the gatherer** with a sharper, narrower prompt (name the specific turn, fork, or contradiction to nail down) and synthesize from its new dossier. Do not "verify" against the source, do not Bash-`cat` the `.jsonl`, do not resolve a discrepancy by investigation — resolving discrepancies is the gatherer's reading job, not yours. Your cost must stay proportional to *thinking*, not *reading*.

**Write discipline — write once.** Prepare each file's full content in your reasoning, then emit it in a **single Write**. Do **not** re-read a file you just wrote to verify it, do not re-write it to tweak it, and do not narrate between writes. The content is fully determined by the dossier — there is no judgment that needs a re-read, so the write-then-verify-then-rewrite loop is pure cost: every extra turn re-bills the whole accumulated context. One Write per `sessions/<id>.md`, one per `log.md`, one per `digest.md`. The only `manifest.json` write is the per-session `high_water` stamp in step 5, done once after all gathers complete.

Fill the frontmatter to the [ARTIFACTS.md](ARTIFACTS.md) contract: `title`, a `thread_relevance` note (does it belong, which subtopic), a `gaps` note (what was not read), and **`extracted_by`** — the synthesizer only, format `<model>-<version> <effort>` (e.g. `opus-4.8 high`). It names the synthesizer, not the gatherer, because fidelity tracks the synthesizer's capability; stamping it makes a thread's extractions comparable across runs and ties any rationale-flattening back to the run that produced it. A later re-extraction stamps whatever produced the current file.

The completion criterion: *every* distinct decision, learning, and mistake accounted for, timestamped, and cited — not a representative sample. That demand is what produces uniform, traceable output instead of a vague recap.

## Regenerate the views (worker steps 5–6)

Both `log.md` and `digest.md` are derived from the session files — never hand-maintained. Regenerate them **whole** each run, in a single write each, to the shape in [ARTIFACTS.md](ARTIFACTS.md). The mechanics here; the contents there.

**`log.md`** — merge the *event* buckets from every session file into one list ordered strictly by timestamp. Sessions overlap in time — that is expected; order by timestamp regardless of which session a line came from.

**`digest.md`** — synthesize from the session files (not the log). **Intent**, **Vision**, and the rest follow the digest contract in [ARTIFACTS.md](ARTIFACTS.md); synthesize Intent & Vision *across* sessions in timestamp order rather than transcribing line-by-line, so the digest reflects the latest thinking and how it shifted.
